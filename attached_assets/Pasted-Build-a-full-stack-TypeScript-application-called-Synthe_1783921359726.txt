Build a full-stack TypeScript application called “Synthetic GTM Signal Engine.”

This application recreates, using entirely fictional data, the architecture of a production GTM workflow that combined product behavior, enrichment, GTM research, a changing growth hypothesis, CRM records, and AI-generated outreach.

The application must simulate the upstream data and decisioning layers. Attio will remain the external CRM and system of record for final people, companies, signals, and generated email records.

Technology:

* Node.js 20+
* TypeScript
* Express
* React
* SQLite
* @faker-js/faker
* Zod
* No authentication required
* Fixed Faker seed for reproducible demo data

The application must contain six modules:

1. Synthetic Enrichment
2. Synthetic Product Analytics
3. Behavioral Trail Builder
4. GTM Research and Prioritization
5. Growth Hypothesis Configuration
6. Attio Export Preparation

ARCHETYPES

Create realistic weighted event sequences for:

1. Rapid Team Activator
    Creates an application within 24 hours, enables an organization, invites teammates, and returns repeatedly.
2. Enterprise Evaluator
    Creates an application, enables an organization, views SSO and enterprise documentation, and visits pricing.
3. Solo Builder
    Creates one application and API key but shows little team or enterprise behavior.
4. Stalled Implementer
    Creates an application, encounters errors, repeatedly views documentation, and becomes inactive.
5. Returning Evaluator
    Initially becomes inactive, then returns, creates or configures an application, and views pricing or SSO resources.
6. Converted Account
    Activates, collaborates, evaluates advanced features, and starts a subscription.

BEHAVIORAL TRAIL

Preserve raw events.

Build a chronological human-readable trail such as:
“July 7: Signed up”
“July 7: Created first application 12 minutes later”
“July 8: Enabled an organization”
“July 8: Invited three teammates”
“July 10: Viewed SSO documentation twice”

Generate a two-to-three sentence behavior summary.

The behavior summary must:

* Use only observed events
* State time from signup to activation
* Mention organization or application creation
* Mention collaboration behavior
* Mention enterprise or pricing research
* Mention errors or inactivity
* Avoid asserting intent as fact
* Use terms such as “may indicate” or “suggests”
* Clearly distinguish observed evidence from interpretation

GROWTH HYPOTHESIS

Create an editable Growth Hypothesis page.

Default hypothesis version: v1.0

Default rules:

* Application creation within 24 hours is a strong activation signal.
* Organization enablement is a stronger team signal than application creation alone.
* Inviting two or more teammates indicates collaboration or expansion.
* Repeated SSO documentation views may indicate enterprise requirements.
* Pricing views without activation are weak intent signals.
* Integration errors followed by inactivity increase churn risk.
* Returning after inactivity may justify assistive outreach.
* Subscription start represents conversion.

Allow users to edit signal weights and save a new hypothesis version.

When the hypothesis changes:

* Preserve previous versions
* Recalculate research assessments
* Show which scores changed
* Keep raw behavior unchanged
