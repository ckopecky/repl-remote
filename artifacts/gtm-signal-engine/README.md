# Synthetic GTM Signal Engine

> ⚠️ **This is a demo built entirely on synthetic, fictional data.** No real companies,
> people, emails, or product usage are represented anywhere in this application. All
> names, companies, domains, and behavioral events are generated by
> [Faker](https://fakerjs.dev/). Nothing in this app calls or writes to Attio, or any
> other real CRM/GTM system — the "Attio Export" screens only render a *preview* of the
> payload shape that would be sent, entirely client/server-side.

## What this demonstrates

A synthetic recreation of a go-to-market signal pipeline that a growth/RevOps team might
build to turn raw product analytics into prioritized, research-backed outbound outreach:

1. **Enrichment** — synthetic companies and people with firmographic/persona data.
2. **Product analytics** — a realistic, per-person event timeline drawn from a fixed
   taxonomy of product events.
3. **Behavioral trail building** — raw events are turned into a human-readable
   chronological narrative plus a short behavior summary.
4. **GTM research & prioritization** — a scoring engine turns the behavioral trail into
   seven 0–100 research scores and an outreach priority bucket.
5. **An editable growth hypothesis** — the weights and messaging guidance behind the
   scoring engine are versioned and editable in the UI; editing and saving triggers a
   full recalculation with a before/after diff.
6. **Attio export prep** — outreach packages render a preview of the exact Company /
   Person / Generative AI Email payloads that would be pushed to Attio. Attio itself is
   never called.

## Architecture

This app substitutes the project's standard PostgreSQL + Drizzle ORM stack for the
SQLite database referenced in the original product spec, since this workspace's
convention is a shared Postgres instance managed through Drizzle. Everything else
(entities, event taxonomy, scoring rules, screens) follows the original spec.

```
artifacts/gtm-signal-engine/   React + Vite frontend (Dashboard, Prospects, Prospect
                                Detail, Growth Hypothesis, Outreach Queue, Demo Controls)
artifacts/api-server/          Express 5 API
  src/routes/                  dashboard, prospects, hypotheses, outreach, demo routers
  src/lib/gtm/
    constants.ts                archetypes, product event taxonomy, outreach statuses
    generator.ts                Faker-based company/person/event generation per archetype
    trail.ts                    raw events -> chronological trail + behavior summary
    scoring.ts                  behavioral trail + hypothesis weights -> research scores
    hypothesis.ts                default growth hypothesis (v1.0) weights & guidance
    seed.ts                     orchestration: reset/reseed/simulate/recalculate
    attio.ts                    Attio-shaped export payload builder (preview only)
lib/db/                        Drizzle schema (7 tables) shared by the API
lib/api-spec/                  OpenAPI spec (source of truth for the contract)
lib/api-zod/, lib/api-client-react/   Generated Zod schemas + React Query hooks
```

## Data model

| Table | Purpose |
|---|---|
| `companies` | Synthetic firmographic data (industry, employee count, funding, ICP fit score) |
| `people` | Synthetic contact enriched with persona/seniority/purchase-role and an assigned **archetype** |
| `product_events` | Raw event stream per person, drawn from the fixed event taxonomy below |
| `behavioral_trails` | One current row per person: a chronological narrative + short behavior summary derived from events |
| `research_assessments` | One current row per person: the 7 scores + outreach priority + rationale, tagged with the hypothesis version that produced them |
| `growth_hypotheses` | Full version history of scoring weights + messaging guidance (never overwritten — a new version is inserted on every edit) |
| `outreach_packages` | Generated outreach records queued for review/approval, holding the exact fields an Attio "Generative AI Email" record would need |

`archetype` was added to `people` beyond the original spec's field list because both the
event generator and the UI need a stable label for "what kind of prospect is this,"
and it doubles as the ground-truth signal the scoring engine is trying to recover from
raw behavior.

## Event taxonomy

```
user_signed_up, application_created, application_configured, organization_enabled,
teammate_invited, sdk_installed, api_key_created, integration_error,
documentation_viewed, sso_documentation_viewed, mfa_enabled, pricing_page_viewed,
enterprise_page_viewed, checkout_started, subscription_started, inactive_period,
returned_to_product
```

## Archetypes

Each synthetic person is generated against one of six behavioral archetypes, which
drives the shape of their event timeline:

- **Rapid Team Activator** — creates an application within 24h of signup, enables an
  organization, invites teammates, and returns repeatedly.
- **Enterprise Evaluator** — creates an application, enables an organization, views SSO
  and enterprise documentation repeatedly, and visits pricing.
- **Solo Builder** — creates one application and an API key with little team/enterprise
  behavior.
- **Stalled Implementer** — creates an application, hits integration errors, repeatedly
  views documentation, then goes inactive.
- **Returning Evaluator** — goes inactive early, then returns and re-engages with
  pricing/SSO research.
- **Converted Account** — activates, collaborates, evaluates advanced features, and
  starts a paid subscription.

## Scoring model

The product spec defines eight qualitative default rules but leaves the exact scoring
formulas open. This implementation (`src/lib/gtm/scoring.ts`) makes them concrete as
follows. All seven scores are 0–100, purely a function of a person's raw events plus the
current growth hypothesis's signal weights (0–1 each) — never of static labels.

| Score | Derived from |
|---|---|
| ICP Fit | Company's synthetic `icpFitScore` (firmographic fit, independent of behavior) |
| Persona Fit | Contact seniority (Individual Contributor → C-Level fit curve) |
| Activation | Application-within-24h (rule 1) + organization enablement (rule 2) |
| Collaboration | Teammate invite count, with a bonus once the 2+ threshold (rule 3) is crossed |
| Enterprise Intent | Repeated SSO documentation views (rule 4), boosted by enterprise/pricing page views and company size |
| Purchase Intent | Pricing views (weak, rule 5) → checkout started (stronger) → subscription started (strongest, rule 8) |
| Churn Risk | Integration errors followed by inactivity (rule 6), offset by returning after a dormant period (rule 7) |

A **composite score** is the mean of the six positive scores minus 0.3× churn risk, then
bucketed into an outreach priority:

- **High** — composite ≥ 50
- **Medium** — composite ≥ 30 (also forced for any account that already converted —
  `subscription_started` — since expansion motions, not new-business urgency, apply)
- **Low** — composite ≥ 12
- **Suppress** — composite < 12, or a hard override when churn risk ≥ 70 **and**
  purchase intent < 20 **and** composite < 40 (a clearly disengaged, low-intent account)

Recommended outreach angle, rationale, and risk notes are generated as short templated
narratives from the same underlying signals, using hedged language ("may indicate",
"suggests") rather than asserted fact, since this is inference over synthetic behavior,
not verified intent.

## Growth hypothesis versioning

`growth_hypotheses` keeps full version history (default `v1.0`, "Activation-Led Growth
v1"). Editing weights or messaging guidance and saving:

1. Inserts a new version (`v2.0`, `v3.0`, ...) and marks it current.
2. Immediately recalculates every person's research assessment against the new weights.
3. Returns a diff of every prospect whose priority or composite score changed, shown to
   the user before they leave the editor.

The `known_limitations` field on every hypothesis version documents that weights are
hand-tuned heuristics, not learned from real conversion data — a way of keeping the
synthetic/demo framing honest even inside the hypothesis UI.

## Attio export

Outreach packages hold the exact fields an Attio "Generative AI Email" record needs:
person/company relationships, campaign, source signal, the full behavioral trail,
behavior summary, research summary, outreach angle, hypothesis version, and prompt
version, with `status = "Ready for Generation"`. `GET
/api/outreach-packages/:id/attio-export` renders this as three separate payload previews
(Company / Person / Generative AI Email) shaped like Attio's `{ objectSlug, values }`
record format. **No network call to Attio is ever made** — this is preview-only, so the
demo can showcase what a real integration would send without any live GTM system
dependency.

## Demo Controls

Available from the slide-over in the left nav, backed by `/api/demo/*`:

- **Generate one prospect** — pick an archetype, get one fully-scored company + person.
- **Generate a batch** — bulk-generate N prospects across random archetypes.
- **Simulate N days** — extends existing people's event timelines with plausible
  continuation events for their archetype (e.g. an Enterprise Evaluator gets more SSO
  doc views; a Stalled Implementer accrues more inactivity) and rescoring everyone.
- **Recalculate** — re-run scoring for every person against the current hypothesis
  without changing the hypothesis itself.
- **Reset demo data** — wipes and regenerates from a **fixed seed** (`20260101`), so the
  starting dataset is reproducible.
- **Reseed demo data** — wipes and regenerates from a **new random seed** each time, for
  a fresh demo dataset.

On first boot with an empty database, the app automatically seeds itself (fixed seed,
~24 people across mixed archetypes and a handful of companies) so the app is never
empty.
